Keyboard Inputs

Designed with context-driven keyboard navigation in mind (kind of the whole point of the application). Turn off your Vimium and follow the bindings wrapped with braces [X]. Context-driven meaning that if some sub-menu is open, only keys from this menu will be valid next inputs; Anything else will reset the context. You can view the current key combination in the bottom right; Where *base* indicates that you have no context. The buttons work too, but don't be expecting a flawless experience if you're jumping from mouse interactions to keyboard interactions.

Suite Types

A keyboard layout's effectiveness goes beyond being able to just program. There is OS navigation, File Navigation, Web Searches and other common actions that a keyboard centric developer may want to test out. In the "Problem Set", you will find a range of different tests:

Type Description
Code Coding challenges are simple typing tests, but with programs instead of random words. New line characters are explicitly defined. Beware, word wrap is on.
Action Action challenges require you to follow the instructions on screen, and enter in commands perfectly. Invalid inputs will reset the line. Completed lines may have some action associated with them, which will usually be explained in the comment above, but will also be be indicated as done with an additional comment added to the line.

Caching Tests

Down the bottom of your screen is a "Cache" checkbox for the local storage of test data. By unchecking this box, you will clear all local storage, and make requests for test data upon each new test. While the checkbox is checked, tests will only make a request on first load. You can "refresh" the tests by simply unchecking and rechecking this checkbox.

Whilst I would love to have a larger array of programming languages on offer, it's important to remember the point of the Code typing tests; To ensure common character sequences can be easily typed. The current array of languages on offer provide a pretty good range of possible character combinations. Additional programming languages should only be added if:
1) Certain character combinations cannot be achieved in existing languages, and
2) Said character combinations are frequent in more than one language.
This just ensures easier maintenance over time.
