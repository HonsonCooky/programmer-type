Keyboard Inputs

This application is designed with context-driven keyboard navigation. Remember to disable Vimium and follow the key bindings indicated by braces [X]. Context-driven navigation meaning that when a sub-menu is open, only keys from that menu will be valid for the next inputs; any other key will reset the context. You can view the current key combination in the bottom right corner, where *base* indicates no active context. While the buttons are functional, switching between mouse and keyboard interactions may not provide a seamless experience.

Future Developer Note: This style of keyboard input indicator is very basic. I've implemented "which-key" style navigation before ( [B] Browser Home ), but it seems a little excessive here as there's at most, two layers to any menu system. Should the need arise, this is an option.

Suite Types

For keyboard centric developers, a keyboard layout's effectiveness goes beyond being able to enter a sequence of English characters. There is OS navigation, File Navigation, Web Searches, Special Characters and other common components that need to be considered and practiced.

Code Suites

Coding challenges are typing tests as you've seen them before. Attempt to correctly type out the sequence of characters on screen as fast as you can. The twist - character sequences are code snippets. The idea here is to test/practice your ability to use common programming character sequences. With that in mind, the following constraints have been implemented:

  • Indentation: Automatic Indentation has been implemented. Each line will start at the first significant character.
  • New-Line Characters: For UX reasons (font-size, screen-size, word-wrap) new line characters are explicitly defined by \n. You only need press Enter when encountering this symbol.
  • No Auto-Complete: Your daily programming setup probably has some more tools that make you blazingly fast. That's not really what we are testing here though.
  • Limited Languages: Your favourite language isn't in the mix?! OUTRAGEOUS! Keep in mind that we are testing/practicing character sequences. The available programming languages aim to cover a wide range of character sequences using as few test suites as possible. Less maintenance ü§∑.

Action Suites

Action challenges are more complicated tests that require you to follow the instructions on screen. This will likely require you to jump to another program or open a new tab. Programmer Type does not track your activity outside of this web application, so completing tasks as instructed will be up to you.

Where required, you might be asked to simulate keyboard inputs from another program. Unlike Coding Challenges, incorrect inputs in these cases will reset the entire line. Accuracy is required here.

Actions might require extra permissions from this web app. Access to your clipboard or file system might be required. Just bare this in mind as your browser might ruin your first attempt by asking for these permissions.

Caching Tests

Down the bottom of your screen is a "Cache" checkbox for the local storage of test data. By unchecking this box, you will clear all local storage, and make HTTP requests for every new test. If you tick the checkbox, test files will be stored in local storage upon first contact. If you find yourself constantly hit with a loading screen between tests, or have the storage on your device to reduce requests to the server, consider checking this box to reduce wait times.
